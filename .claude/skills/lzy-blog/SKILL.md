# lzy-blog 写作规范

用 lzy 本人的口吻写技术博客。核心一句话:直接解释,能省则省。

## 博客格式(硬性)

- 每篇必须有 frontmatter,格式如下,`date` 精确到秒:

```yaml
---
title: 文章标题
date: 2026-07-09 10:10:00
categories: 技术栈
tags:
    - Java
---
```

- `date` 的时刻集中在 `22:00:00` 到次日 `03:00:00` 之间(深夜时段),分秒随机生成,不要整点。跨零点时日期顺延一天。
- 系列文章的 `date` 要按篇号递增,前一篇一定早于后一篇,保持前后次序。
- 正文不写一级标题 `#`。标题由 frontmatter 的 `title` 承担,frontmatter 之后直接进正文或从 `##` 二级标题起。
- 文件放在 `docs/posts/<主题>/` 下,系列文章用 `1-xxx.md`、`2-xxx.md` 顺序命名。
- 单行超过 100 列的要换行,正文和代码注释都适用。
- 正文续行用行尾两个空格(Markdown 硬换行)接到下一行,不要用空行。空行只用于真正的话题分段或标题、代码块前后。同一段解释的多行连续排布,行尾都留两个空格,最后一行不留。

## 直接解释是第一要务

- 上来就讲这个东西是什么、加了之后有什么效果、怎么做到。不铺垫、不预告、不总结自己要讲什么。
- 每句必须有信息量。禁止空话废话句,例如"只做准备,不启动""为后续铺好料""这一步很关键""下面来看"——读完不增加任何信息的句子一律删掉。
- 讲清效果和实现,而非只给结论。出现生僻或前置概念时补一句等价物(如 `@Bean` → Spring 中手动注册 Bean 的注解,等价于早期 XML 的 `<bean>`)。

## 能省则省

- 删掉一切不承载信息的字。宁短勿长。
- **禁止破折号 `——`**。需要转折或补充,直接用句号断句,或另起一行,或拆成标题。
- **克制加粗**。加粗只留给真正的关键术语,不用加粗当小标题、不每段开头都加粗。
- **少用冒号**。避免"结论:解释"这种句式堆砌,直接写成完整句子。

## 主线

- 每篇围绕一条主线展开。开头就用一句话点明本篇要讲清的问题,让读者一进来就知道读完能答上什么。例:「引入一个依赖就自动配好一堆 Bean,到底靠什么做到」。
- 这句话直接写成陈述句作为开场,不加「本篇要讲清」「本文将介绍」这类元开场白。
- 点明问题后,紧跟一句说明用什么方式展开,并全篇顺着走:先总后分、层层展开、或按步骤推进,选一种,不跑题。
- 章节之间要有联系。上一节结束时带出下一节,或下一节开头承接上一节。不从一个概念直接蹦到另一个,中间用一句桥接说清怎么走过来的。
- 讲清当前知识点后,指出它又引出哪个点,把接力棒传给下一篇。

## 结构:标题切分 + 及时换行

- 用二级/三级标题切分知识点,标题本身就是结论。不用加粗句子冒充小标题。
- 及时换行、及时分段。句子短,一段讲一件事,不堆长段落。
- 整篇层层递进,每个点是上一点的自然延伸,不做并列散点罗列。
- 收口埋钩子,点出下一个知识点要解决的问题。

## demo

- 每个知识点都配代码 demo。
- demo 直接给,不写文字铺垫。
- **凡是摘录框架或库的源码,代码块首行必须用 `// 源码` 标明**,可带定位(如 `// 源码 SpringApplication#run`)。自己写的示例代码不加此标,按场景用 `// 传统写法`、`// Boot写法` 等标注。读者要能一眼分清哪段是框架源码、哪段是示例。
- 注释描述作用或效果。

## 用词禁忌

- 不出现人称代词:你、我、他、它。指代某个类/注解/概念时直接写名称。
- 不用成语。
- 不用官方腔的板块标签(痛点对照、机制拆解、落点、反向验证、隐藏结论、观测手段、记住 等)。要下结论就直接下。
- 有确切的专业技术名词就直接用,不自造含糊说法。例:该叫「生命周期回调」就不写「沿途的扩展点」,该叫「组件扫描」就不写「自动找类」。第一次出现时补一句解释即可。

## 关联日常代码

- 能对应到项目里高频写法的原理,就落到那段代码上,不停在抽象机制。两种情形:
  - 这个机制本身就是平时常写的代码。例:启动后初始化任务就是 `CommandLineRunner`。
  - 平时写的某段代码,底层正是这个机制。例:改 `server.port` 能生效,靠的是内嵌容器创建那一步。
- 示例用真实高频场景(预热缓存、上报注册中心、改端口),不造抽象的 `foo/bar` Demo。让读者把原理和自己写过的代码对上号。

## 收口

- 技术类主题落到具体的注解、包内接口或扩展点;其他主题落到可操作的结论。
